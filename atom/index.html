<p>最近周围好多人都换成机械键盘，甚至也有HHKB。我试着把玩了几下，手感较薄膜键盘还是 有些提升的，加上更适合码农的键位，看着是挺诱人的。要是以前的我，心中必定一地鸡 毛，然后一通调研比较各种牌子价位，最后因为囊中羞涩而作罢，时间也浪费掉了。而现在 我学的淡定了许多。从我个人分析，键盘作为码农的生产工具，换机械键盘的原动力主要有 两个：</p> <ol> <li>键盘好看。87键位小巧可爱，与众不同。</li> <li>号称适合码农。良好的手感，<code>Caps</code> 和 <code>Caps</code> 可以互换。</li> </ol> <p>原因1无可厚非，等同于看到美女时的视觉享受，无法抗拒呀，:)。原因2有个深层的假设， 高手们都在用机械键盘，或许当我也换成使用这种键盘时，我的效率就会自然而然得到提高， 不假时日也能进入高手的行列。很显然，想通过换键盘来提高rank的行为相当自欺欺人，而 这恰恰是换机械键盘时的最大幻想。在试玩的过程中由于键位的不适应，刚开始使用时效率 反而会比我以前惯用的键盘低很多。可见，键盘无所谓贵贱，“唯手熟耳”。至于美感的问题， 这主要还是取决于你的钱包厚度。</p> <p>其实困扰低级码农的工具问题不止键盘，各种工具的选择都会对低级码农造成不小的麻烦： 哪种语言更高级？编辑器和<code>IDE</code>的哪个好？<code>vim</code>和<code>emacs</code>哪个更好？<code>IDE</code>哪个更好用？ 对于处于并将很长期处于低级码农的我来说，这些问题浪费了我很多时间和精力，更可怕的 是因为纠结于这些问题，忽略了你真正要完成的目的。而且这些问题本来就是没有答案的， 今天看到那个说A好就玩玩A，明天看到哪个说B好就转向B，到头来都是一场空。所以不要纠 结在这些形而上的东西，能候选的肯定都不会差，选中一个就可以开始你的精彩之旅。</p> <p>现实是人类总是很愚蠢的，我又开始“关心”哪种方式写erlang最方便。在浪费了一个礼拜的 “选择”后，我最终觉得<code>erlang</code>还是用已经成为<code>erlang</code>届既定标准的<code>emacs</code>来写比较好， 毕竟排版最好看，再配合<code>evil</code>就更好了。至于erlIDE或者IDEA的erlang插件，想要花时间 再去学，还是把这个时间写更多的代码呢？:)</p> <p>但愿以后多点智慧。</p> <ol> <li>remote call/external function call: <code>m:f(e1, ..., eN)</code></li> <li>local call: <code>f(e1, ..., eN)</code></li> </ol> <p>remote call 和 local call的主要差别在热更新时。erlang运行时对module可以维护 两个版本的代码:<code>current</code>和 <code>old</code>。老版本和新版本可以同时运行：remote call总是使用 <code>current</code>版本，local call可能运行<code>old</code>版本。当有更新的版本代码加载时，运行<code>old</code> 代码的进程会终止。可以通过调用remote call，使进程切换到 <code>current</code> 版本。</p> <pre class="highlight"><code>-module(m). -export([loop/0]). loop() -&gt; receive code_switch -&gt; m:loop(); Msg -&gt; do_something, loop() end.</code></pre> <p>这种差别也体现在spawn函数调用时，<code>spawn(M, F, A)</code> 使用<code>current</code>代码，而 <code>spawn(Fun)</code> 中 <code>Fun</code> 是local call，不能进行热更新。 </p> <p>但是存在这样一个问题：当进行热更新时，怎么保证运行老版本代码的进程不死掉呢？</p> <h4>---
